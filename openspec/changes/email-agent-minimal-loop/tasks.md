## 当前阶段执行看板

下面这部分用于约束“此刻应该做什么”，避免已经有基础骨架后继续在外围能力上失焦。

### 当前主目标

当前阶段的唯一主目标是：

- 把“邮件智能客服的 AI 处理主链路”做成可诊断、可演示、可复盘的最小闭环

这里的“AI 处理主链路”特指：

- 意图归一化
- 场景与子意图路由
- 上下文加载与压缩
- 业务 facts 与知识检索协同
- 回复草稿生成
- 审核与兜底

- [x] 明确意图识别的“启发式基线 + 模型增强 + 回退策略”行为，并保留调试证据
- [x] 明确上下文压缩触发条件、摘要来源、覆盖消息数与回退逻辑

- [ ] 补齐人工审核、追问、阻塞这三类兜底路径的演示与验证证据
- [ ] 沉淀一份典型误判 / 踩坑记录，说明当前版本容易出错的位置与修正方式

### 当前明确暂缓的任务

下面这些任务不是当前阶段的主矛盾，除非影响最小闭环跑通，否则暂不继续深化：

- [ ] 深挖 `IMAP` 细节能力
- [ ] 深挖 `SMTP` 自动发送策略
- [ ] 增加更多队列管理页面或外围运维视图
- [ ] 提前做多渠道接入
- [ ] 提前接入复杂工单流转
- [ ] 为未来假设场景做通用平台化抽象

### 停止条件

如果某项工作满足下面任一条件，默认判定为“先停下来，回到 OpenSpec 边界重新确认”：

1. 主要收益落在渠道、运维或外围控制面，而不是 AI 主链路
2. 改动不能直接提升“意图 / 上下文 / RAG / 草稿 / 兜底”的可演示性
3. 需要引入较重的新抽象，但当前没有真实重复场景支撑
4. 为未来可能需要的能力提前铺路，且当前没有验证闭环价值
5. 用户无法通过一次 demo 明确看出这项改动改善了什么

### 执行顺序

当前阶段按下面顺序推进：

1. 先补 AI 主链路的诊断与可观测性
2. 再补意图、上下文、RAG 的具体质量改进
3. 再补审核与反馈闭环
4. 最后才回头补真实收发信联调或更重的外围能力

### 完成判定

当前阶段不以“真实 IMAP/SMTP 全接通”作为完成标准，而以以下证据作为完成判定：

1. 能跑通至少一个售前样例和一个售后样例
2. 能说明每一步 AI 判断为什么这样做
3. 能展示 facts 与 knowledge 分别提供了什么信息
4. 能展示追问、人工审核、阻塞三类兜底路径
5. 能留下可复盘的输入、输出、日志与诊断结果
