# 规格：邮件 Agent 最小闭环

## 概述

第一阶段系统必须实现一条公共邮件 agent 主链路，在共享主流程上分别支持售前和售后两类处理分支。

## 需求

### R1. 邮件接入

第一阶段系统必须以客户来信邮件作为主入口渠道。

第一阶段流程必须从以下步骤开始：

- 邮件接入
- 内容清洗
- 提取当前回合最新且可行动的客户问题

### R2. 意图重写层

在最终意图映射之前，系统必须包含一层“意图重写 / 归一化”处理。

这层必须：

- 把带噪声的自然语言邮件整理成更清晰的内部请求表达
- 提升后续意图分类或检索质量
- 不得凭空编造邮件和上下文中不存在的事实

第一阶段这层必须至少输出以下结构化结果：

- 归一化后的内部请求表达
- 客户当前主问题
- 附带问题列表
- 一级场景候选
- 子意图候选
- 关键必需实体
- 缺失实体
- 当前可处理状态：直接继续 / 先追问 / 人工介入

第一阶段这层不得改写以下关键信息：

- 订单号
- 物流单号
- SKU 或产品标识
- 日期、金额、数量、地址
- 客户明确表达的风险诉求或承诺相关诉求

第一阶段系统必须支持把“信息不足但可补问”的情况，与“高风险需人工介入”的情况明确区分开。

### R3. 意图路由

系统必须先将请求路由到两个一级场景：

- 售前
- 售后

第一阶段系统必须支持在每个一级场景下继续承载多个子意图。

第一阶段系统不得把“两个一级场景”误等同于“只有两个意图”。

第一阶段必须先收敛一份首批子意图清单，作为最小闭环的初始覆盖集。

当前首批子意图建议至少包括：

- 售前：产品推荐、产品差异说明、基础功能咨询、发货或库存咨询
- 售后：订单状态查询、物流进度查询、常见售后政策解释、标准化售后问题回复

### R4. 上下文加载

第一阶段系统必须使用可用的历史邮件作为上下文。

如果可用，系统还应该使用画像或行为信号，例如：

- 历史订单
- 浏览记录
- 购物车状态

系统必须利用上下文尽量减少让客户重复提供已有信息。

第一阶段必须区分：

- 必需上下文
- 强相关增强上下文
- 可选画像增强

当前约束：

- 历史邮件线程摘要属于必需上下文
- 订单、物流、售后处理摘要属于强相关增强上下文
- 浏览记录、购物车、偏好标签属于可选画像增强
- 可选画像缺失不得阻塞主链路

当上下文互相冲突时，系统必须优先采用：

1. 实时业务系统事实
2. 当前回合客户明确表达
3. 当前线程最近上下文
4. 较旧画像或行为信号

### R5. 售前处理

对于售前邮件，第一阶段系统至少必须支持以下能力：

- 产品推荐
- 产品差异说明
- 基础功能咨询
- 发货或库存咨询

在做产品推荐时，系统应该优先给出结构化、多候选的建议，而不是单点拍板式答案。

第一阶段售前分支还必须满足：

- 推荐结果应优先给出 2 到 3 个有差异的候选方向
- 应说明当前推荐依据与未确认条件
- 信息不足时应优先进入最小追问，而不是武断推荐
- 不得把规则级说明伪装成实时库存或履约真值

### R6. 售后处理

对于售后邮件，第一阶段系统至少必须支持以下能力：

- 订单状态查询
- 物流进度查询
- 常见售后政策解释
- 标准化售后问题回复生成

当业务数据可用时，回复必须优先体现：

- 当前事实状态
- 下一步动作
- 预期后续路径

第一阶段售后分支还必须满足：

- 缺少订单号、物流号或无法唯一定位时，应优先进入追问
- 数据冲突、异常状态、高风险诉求时，应支持转人工或人工审核
- 政策说明可以补充事实解释，但不得覆盖业务真值

### R7. 外部业务数据与 RAG

系统必须支持查询订单、物流等业务事实数据。

系统必须支持基于 RAG 的产品、规则或政策知识补充。

系统必须把业务系统事实置于模型自由生成之上。

RAG 不得替代权威的订单或物流数据。

第一阶段系统必须通过统一业务数据接入边界查询订单、物流和标准售后规则能力。

第一阶段业务主流程不得直接绑定以下任一具体接入形式：

- MCP 协议细节
- 内部 API SDK 细节
- skill 原始输出
- CLI 原始输出

第一阶段业务数据查询层至少必须能区分以下结果状态：

- 查询成功
- 查询条件不足
- 无查询结果
- 临时失败
- 结果冲突

第一阶段 RAG 只应覆盖以下知识范围：

- 产品知识
- 售前基础说明
- 常见售后政策说明
- FAQ 或标准问答素材

第一阶段 RAG 不应承担以下职责：

- 订单真值查询
- 物流真值查询
- 高风险承诺决策

如果同一封邮件同时需要业务事实和知识补充，系统必须优先：

1. 获取业务 facts
2. 再补充知识 explanation
3. 生成回复时先呈现事实，再补充说明

### R8. 多问题邮件

系统必须假设一封邮件可能同时包含多个问题。

第一阶段设计至少必须能区分：

- 主问题
- 附带问题

第一版实现可以只优先处理主问题，但这种行为必须在后续实现规格里写清楚。

### R9. 回复草稿生成

第一阶段系统必须以“回复草稿”作为主要输出物。

第一阶段能否直接自动发送，不作为闭环完成的前置条件。

第一阶段输出至少必须能区分三类状态：

- 可直接生成回复草稿
- 需要先追问补充信息
- 需要进入人工处理或人工审核

### R10. 审核与兜底

在最终发送策略完全确定前，第一阶段系统必须保留校验、审核或人工接管节点。

系统必须在以下场景支持兜底：

- 置信度过低
- 缺少关键实体
- 业务数据与生成内容冲突
- 问题高风险或需要人工判断

第一阶段审核或兜底层至少必须能区分：

- 可放行
- 需要修改或补充
- 需要人工审核
- 需要阻塞处理

第一阶段回复状态至少必须显式体现：

- `draft_ready`
- `follow_up_needed`
- `human_review_required`
- `blocked`

第一阶段还必须保留最小结构化事件链，用于回放以下关键节点：

- 邮件接入
- 意图归一化
- 场景与子意图判断
- 上下文加载
- 业务查询
- RAG 检索
- 草稿生成
- 审核或兜底决策

### R11. 第一阶段边界

第一阶段系统不要求：

- 多渠道支持
- 全量意图覆盖
- 复杂工单流转
- 锁定最终发送模式
- 对高风险客服场景做全自动处理
