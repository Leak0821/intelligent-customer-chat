# 规格：邮件 Agent 最小闭环

## 概述

第一阶段系统必须实现一条公共邮件 agent 主链路，在共享主流程上分别支持售前和售后两类处理分支。

## 需求

### R1. 邮件接入

第一阶段系统必须以客户来信邮件作为主入口渠道。

第一阶段流程必须从以下步骤开始：

- 邮件接入
- 内容清洗
- 提取当前回合最新且可行动的客户问题

### R2. 意图重写层

在最终意图映射之前，系统必须包含一层“意图重写 / 归一化”处理。

这层必须：

- 把带噪声的自然语言邮件整理成更清晰的内部请求表达
- 提升后续意图分类或检索质量
- 不得凭空编造邮件和上下文中不存在的事实

### R3. 意图路由

系统必须先将请求路由到两个一级场景：

- 售前
- 售后

第一阶段系统必须支持在每个一级场景下继续承载多个子意图。

第一阶段系统不得把“两个一级场景”误等同于“只有两个意图”。

### R4. 上下文加载

第一阶段系统必须使用可用的历史邮件作为上下文。

如果可用，系统还应该使用画像或行为信号，例如：

- 历史订单
- 浏览记录
- 购物车状态

系统必须利用上下文尽量减少让客户重复提供已有信息。

### R5. 售前处理

对于售前邮件，第一阶段系统至少必须支持以下能力：

- 产品推荐
- 产品差异说明
- 基础功能咨询
- 发货或库存咨询

在做产品推荐时，系统应该优先给出结构化、多候选的建议，而不是单点拍板式答案。

### R6. 售后处理

对于售后邮件，第一阶段系统至少必须支持以下能力：

- 订单状态查询
- 物流进度查询
- 常见售后政策解释
- 标准化售后问题回复生成

当业务数据可用时，回复必须优先体现：

- 当前事实状态
- 下一步动作
- 预期后续路径

### R7. 外部业务数据与 RAG

系统必须支持查询订单、物流等业务事实数据。

系统必须支持基于 RAG 的产品、规则或政策知识补充。

系统必须把业务系统事实置于模型自由生成之上。

RAG 不得替代权威的订单或物流数据。

### R8. 多问题邮件

系统必须假设一封邮件可能同时包含多个问题。

第一阶段设计至少必须能区分：

- 主问题
- 附带问题

第一版实现可以只优先处理主问题，但这种行为必须在后续实现规格里写清楚。

### R9. 回复草稿生成

第一阶段系统必须以“回复草稿”作为主要输出物。

第一阶段能否直接自动发送，不作为闭环完成的前置条件。

### R10. 审核与兜底

在最终发送策略完全确定前，第一阶段系统必须保留校验、审核或人工接管节点。

系统必须在以下场景支持兜底：

- 置信度过低
- 缺少关键实体
- 业务数据与生成内容冲突
- 问题高风险或需要人工判断

### R11. 第一阶段边界

第一阶段系统不要求：

- 多渠道支持
- 全量意图覆盖
- 复杂工单流转
- 锁定最终发送模式
- 对高风险客服场景做全自动处理
