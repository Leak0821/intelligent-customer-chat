# 提案：邮件 Agent 最小闭环

## 背景

当前项目已经具备：

- 仓库治理规则
- AI 工作流规则
- 工程实现原则
- 第一阶段项目范围定义
- 推荐采用通用主链路 `C` 的研究结论

但项目还缺少一份正式变更合同，用来定义第一条邮件 agent 最小闭环。

## 问题

当前 `project-scope` 已经说明第一阶段覆盖什么，但还没有形成一个正式的实现合同，来约束：

- 收到邮件后如何进入系统主流程
- 意图重写应该放在哪一层
- 售前和售后如何分流
- 上下文、外部业务数据和 RAG 如何组合
- 发送前的审核或兜底节点放在哪里

如果缺少这份合同，后续实现很容易范围漂移或不断膨胀。

## 目标

- 定义项目第一条面向生产形态的邮件 agent 闭环
- 采用一条公共主链路，并在中途分出售前和售后分支
- 明确保留意图重写 / 归一化层
- 第一版既要足够小，能真正落地，又要足够完整，能体现真实系统
- 从第一版开始就保留审核、兜底和人工接管扩展位

## 非目标

- 锁定最终技术栈
- 锁定最终 MCP、skill 或 CLI 接入方式
- 冻结完整意图树
- 提前决定最终发送模式是全自动还是全人工审核
- 在第一版里覆盖所有客服场景

## 预期结果

建立第一份正式系统合同，描述一条最小但完整的邮件 agent 闭环：

1. 接收并清洗邮件
2. 对用户问题做重写或归一化
3. 映射到一级场景与子意图
4. 加载上下文和可用的用户画像信号
5. 进入售前或售后处理分支
6. 结合外部业务数据和 RAG 补充知识
7. 生成回复草稿
8. 进入校验、审核或发送前状态

## 为什么现在做

项目现在应该从“规范已经建立”进入“开始按规格驱动”的阶段。

这次变更能带来的价值：

- 形成第一阶段系统目标
- 为后续 OpenSpec 变更建立稳定基线
- 相比单点 demo，形成更完整、更适合面试展示的系统叙事
