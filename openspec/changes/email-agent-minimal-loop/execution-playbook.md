# 执行守则：邮件 Agent 第一阶段最小闭环

## 1. 目的

这份文档不是重新定义需求，而是把当前变更在实现阶段的执行纪律写清楚。

它主要解决两个问题：

- 当前已经有不少设计和代码骨架，后续容易继续往外围能力深挖
- 如果没有一份“每轮到底该做什么”的执行守则，任务会逐渐偏离 AI 主链路

因此，从当前版本开始，这份文档作为 `email-agent-minimal-loop` 的实现期强约束。

## 2. 当前唯一主目标

当前阶段唯一主目标固定为：

- 把邮件智能客服的 AI 处理主链路做成可跑通、可解释、可演示、可复盘

这里的 AI 主链路只包含：

- 意图归一化与路由
- 上下文加载与压缩
- facts 查询与知识检索协同
- 回复草稿生成
- 审核、追问、阻塞、转人工等兜底

不包含：

- 更深的收发信协议打磨
- 更重的队列和运维控制面
- 多渠道平台化抽象
- 为未来能力做过早基础设施铺垫

## 3. 每轮任务选择规则

进入实现阶段后，每一轮只允许选择一个主任务。

主任务必须同时满足下面三个条件中的至少两个：

1. 直接提升最小闭环的可跑通性
2. 直接提升最小闭环的可解释性
3. 直接提升最小闭环的可演示性

如果一个任务主要提升的是：

- 渠道接入完整度
- 运维便利性
- 未来扩展性
- 平台抽象完整度

则默认不进入当前轮次，除非它已经阻塞 AI 主链路继续推进。

## 4. 实现前必答四问

每次开始一个实现任务前，先明确回答下面四个问题：

1. 这个任务改善的是哪一段 AI 主链路
2. 用户在 demo 或回放里能看到什么变化
3. 没做这件事，当前闭环会卡在哪里
4. 做到什么程度就该停，不继续往外扩

如果这四个问题答不清楚，默认不进入实现。

## 5. 当前允许优先推进的事项

当前优先级固定如下：

1. AI 主链路诊断与证据补全
2. 意图识别质量与回退策略
3. 上下文压缩、历史线程利用与可解释性
4. RAG 检索质量、facts/knowledge 协同与证据展示
5. 回复草稿质量与审核、追问、阻塞路径闭环
6. 首批售前 / 售后高频子意图的稳定演示样例

## 6. 当前明确暂缓的事项

下面这些事情不是不能做，而是当前默认不继续深挖：

- `IMAP` 的更细协议兼容问题
- `SMTP` 的真实自动发信放量策略
- 复杂队列调度、抢占、补偿编排
- 更多管理后台页面
- 多渠道统一接入
- 多 agent 平台化编排
- 过细的领域抽象或未来插件化改造

只有在它们已经直接阻塞当前闭环验证时，才允许临时插队处理。

## 7. 偏航识别信号

出现下面任一信号，就判定为偏航：

- 连续几轮都在改外围接入层，但 `analysis / replay / evaluation` 没有更强证据
- 新增了很多接口或状态，但用户仍然看不懂模型为何这样判断
- 某个能力实现深度已经超过当前面试 demo 讲解需要
- 开始讨论或实现未来多渠道、多租户、多系统复用抽象
- 需要写很长说明才能解释“这项改动到底改善了什么”

一旦偏航，处理方式固定为：

1. 停止继续扩当前子任务
2. 回到 [tasks.md](/Users/leak/Documents/code/personal/learning/ai/intelligent-customer-chat/openspec/changes/email-agent-minimal-loop/tasks.md)
3. 把该项标记为暂缓或后置
4. 重新选择一个直接改善 AI 主链路的任务

## 8. 当前版本的完成判定

当前阶段完成，不以“真实生产已接通”作为标准，而以以下证据作为标准：

1. 至少一个售前样例和一个售后样例可稳定跑通
2. 可以说明意图识别、上下文压缩、RAG、草稿生成为什么得到当前结果
3. facts 与 knowledge 的分工能被展示出来
4. 追问、人工审核、阻塞三类兜底路径都能演示
5. 输入、输出、诊断、回放、评估证据可留存

## 9. 当前阶段的任务推进模板

后续每次推进一个子任务时，建议按下面模板执行：

1. 先在 `tasks.md` 对齐当前主任务
2. 明确本轮只改哪一段链路
3. 先补证据和诊断，再补能力本身
4. 做最小实现，不顺手扩周边
5. 以测试、smoke 或 demo 回放作为结束证据
6. 把真实踩坑补进 [docs/development-pitfalls.md](/Users/leak/Documents/code/personal/learning/ai/intelligent-customer-chat/docs/development-pitfalls.md)
