# 设计：邮件 Agent 最小闭环

## 1. 设计目标

第一版要表现得像一个真实客服系统的主干，而不是一条单点演示路径。

这意味着：

- 有共享接入主链路
- 有显式的归一化处理层
- 有按场景分支的处理方式
- 有外部业务数据参与的回复生成
- 从一开始就保留兜底与审核节点

## 2. 采用的结构

本设计采用一条公共主链路 `C`：

```text
邮件接入
  -> 邮件清洗
  -> 意图重写
  -> 一级场景 + 子意图映射
  -> 上下文与画像加载
  -> 进入售前或售后分支
  -> 业务数据查询与/或 RAG 补充
  -> 生成回复草稿
  -> 校验 / 审核 / 发送前状态
```

## 3. 为什么采用这种结构

### 3.1 比单点 Demo 更稳

单点 demo 虽然更容易完成，但在以下场景很容易崩塌：

- 新增意图
- 引入售后支持
- 引入更复杂的业务数据查询

采用公共主链路，可以避免后续每加一个能力就重写一遍系统骨架。

### 3.2 更适合面试展示

这套结构能体现：

- 领域拆解能力
- 分阶段决策能力
- 真实业务数据优先原则
- RAG 只是增强，不是真相源
- 通过审核和兜底体现上线可信度

## 4. 关键层决策

### 4.1 意图重写层必须显式存在

原始邮件往往：

- 噪声很大
- 同时包含多个问题
- 信息不完整
- 情绪化表达较强

因此“意图重写 / 归一化”不应被隐藏成实现细节，而应被视为正式一层。

这样后续才能继续明确：

- 什么内容需要被重写
- 什么内容不能被改写
- 重写质量如何验证

第一阶段这层不应只输出一段自由文本，而应输出稳定结构，至少包括：

- 归一化后的内部请求表达
- 主问题
- 附带问题
- 一级场景候选
- 子意图候选
- 关键实体缺口
- 直接继续 / 先追问 / 人工介入 判断

这份合同的详细版本见：

- `research/first-stage-intent-normalization-contract.md`

### 4.2 先一级场景，再子意图

设计上先判断：

- 是售前？
- 还是售后？

然后再映射到更细的子意图。

这样可以保证分支结构稳定，同时允许意图体系逐步扩展。

第一阶段不应空着子意图范围直接进入实现，而应先固定一份 seed list，作为首批覆盖集。

当前建议的 seed list 见：

- `research/first-stage-sub-intent-seed-list.md`

### 4.3 先加载上下文，再做重推理

系统在进入更深层判断前，应先加载可用上下文。

原因是：一旦忽略历史信息，用户体验会迅速下降。

尤其在以下场景更明显：

- 售后多轮追问
- 老客户重复来信
- 售前推荐链路中的连续追问

但第一阶段不应把所有可能数据都堆成前置依赖。

更稳的做法是分层加载：

- 历史邮件线程摘要作为必需上下文
- 订单、物流、售后记录作为强相关增强上下文
- 浏览、购物车、偏好作为可选画像增强

这样既能保留真实客服体验，又不会因为画像系统不稳定而拖垮主链路。

这份合同的详细版本见：

- `research/first-stage-context-loading-contract.md`

### 4.4 业务事实优先于自由生成

当订单或物流事实存在时，回复应先围绕事实展开。

这对售后尤其关键，因为“礼貌但模糊”的回答，往往比“简洁但明确”的事实回复更糟。

### 4.5 审核节点是主链路的一部分

审核或校验节点不是以后再加的外挂，而是第一版就应保留的结构。

原因包括：

- 最终发送策略尚未定稿
- 高风险回复需要安全检查点
- 面试和真实落地都更看重系统可信度

### 4.6 输出状态必须先明确

如果不先定义主链路的输出状态，后面实现会默认变成“所有请求都直接出答案”。

这不符合真实邮件客服。

第一阶段更合理的做法是，至少区分：

- 可直接生成回复草稿
- 需要追问补充信息
- 需要人工处理或人工审核

## 5. 第一版取舍

### 已接受

- 邮件作为第一阶段唯一渠道
- 只覆盖少量高频子意图
- 画像信号在不可用时可以降级
- 回复草稿作为第一阶段稳定输出物

### 延后处理

- 完整意图树
- 最终发送模式
- 所有高级异常流程
- 广义投诉与补偿处理
- 多渠道编排

## 6. 需要警惕的风险

- 子意图范围过早膨胀
- 如果没有真实样本，意图重写很容易过拟合
- 画像数据在早期可能缺失或质量不稳定
- 如果没有坚持业务数据优先，RAG 会被误用
- 多问题邮件会暴露主问题 / 附带问题的处理缺口

## 7. 第一阶段必须收敛的技术决策

当前主链路已经基本收敛，但在进入实现前，至少还要明确以下技术决策：

1. 邮件接入方式
2. 主流程编排方式
3. 意图重写与意图匹配方式
4. 上下文与画像存储方式
5. 外部业务数据接入方式
6. RAG 第一版落地方式
7. 回复草稿、审核与发送前状态
8. 输出状态分类方式
9. 日志、回放与评估方式

这些点的详细清单见：

- `research/first-stage-technical-decisions.md`

当前设计约束：

- 这些技术决策必须服务于第一阶段最小闭环，而不是为了未来做过度抽象
- 在这些决策没有收敛前，不应直接进入实现任务拆分
- 对这些问题可以做定向 brainstorming，但确认结论后必须回写当前设计

## 8. 当前推荐的技术收敛方向

在不锁死最终技术栈的前提下，当前更推荐以下方向：

- 第一阶段邮件接入先采用 `IMAP`，并通过邮件接入适配层统一收口
- 主流程编排优先采用 `Java + Spring Boot + 显式状态主链路`，而不是先上过重框架
- 意图重写与意图匹配优先采用“两步走 + 结构化输出”
- 在实现前先固定首批子意图 seed list
- 历史邮件作为必备上下文，用户画像作为可选增强
- 订单/物流等业务能力优先通过适配层接入，而不是直接绑死某单一协议
- 第一阶段采用轻量 RAG，只覆盖产品、规则和 FAQ 知识
- 把回复草稿和审核前状态设计成可持久化、可追踪的系统状态
- 把“直接回复 / 先追问 / 转人工”设计成显式输出状态
- 从第一版开始保留事件日志和可回放链路

补充的当前推荐 Java 技术基线：

- 应用骨架：`Spring Boot`
- AI 能力主抽象：`Spring AI`
- 阿里模型生态补充层：按需引入 `Spring AI Alibaba`
- 后续如需更强的异步编排或长事务能力，再单独评估 `Temporal`，不作为第一阶段默认前提

补充说明：

- 当前已确认的第一阶段邮件接入决策为 `IMAP`
- 该决策是第一阶段落地方案，不代表未来最终只能使用 IMAP
- 后续如接入事件推送或第三方平台，应通过同一接入适配层扩展
- 当前已确认的主流程方向是 `Java 显式状态主链路`
- 当前不建议第一阶段直接把业务主流程绑定到 `Spring AI Alibaba` 的特性上
- 更稳的方式是：业务层先依赖统一 AI 抽象，再按 provider 接入 `DeepSeek`、`Qwen` 或 embedding 能力

推荐的模型接入边界：

- 对话模型如 `DeepSeek`，优先通过 `Spring AI` 官方 provider 接入
- 阿里云百炼、`Qwen`、千问 embedding 等能力，优先通过 `Spring AI Alibaba` 作为补充 provider 适配层接入
- 业务流程只依赖内部接口，例如 `LlmClient`、`EmbeddingService`、`ReplyPlanner`

这样设计的目的：

- 保持第一阶段主链路可调试、可替换、可解释
- 避免因为 provider 变化导致主流程重写
- 为后续接入 MCP、RAG、更多模型或审核策略保留干净边界

补充的意图重写实现约束：

- 这一层只负责整理表达，不负责确认业务真值
- 订单号、物流号、SKU、日期、金额、数量等关键实体不得被模型自由改写
- 多问题邮件至少要产出“主问题 + 附带问题”的结构
- 缺少关键实体时优先进入“先追问”，高风险场景优先进入“人工介入”

补充的上下文加载约束：

- 历史邮件线程摘要属于第一阶段必需上下文
- 订单、物流、售后处理摘要属于强相关增强上下文
- 浏览、购物车、偏好标签属于可选画像增强
- 可选画像缺失不得阻塞主链路
- 上下文冲突时，实时业务事实优先于历史摘要和画像信号

补充的外部业务数据接入约束：

- 业务主流程应依赖内部 gateway 或 adapter 抽象，而不是直接依赖 MCP、API、skill 或 CLI 细节
- 第一阶段至少要覆盖订单查询、物流查询、标准售后规则查询三类能力
- 查询结果必须结构化返回成功、条件不足、无结果、临时失败、冲突等状态
- 下游业务系统异常不得被伪装成“无结果”

参考合同：

- `research/first-stage-business-data-access-contract.md`

补充的第一阶段 RAG 约束：

- RAG 只作为产品知识、规则说明、FAQ 的补充层
- 订单状态、物流节点、售后处理进度必须来自业务事实层
- 回复生成时应先使用业务 facts，再补充知识 snippets
- 第一阶段不应把未治理的大量零散文档直接并入知识源
- 第一阶段允许通过受控后台入口手工导入 `CSV / Word / PDF`
- 导入链路应先把文件落成父文档，再按稳定语义边界切成子文档写入 ES
- 子文档必须保留来源文件、场景、子意图和 `parent_id`，方便后续检索归因与回放
- 父文档必须带 `knowledge_key / version / status / content_hash`
- 子文档必须带 `knowledge_key / version / status / chunk_hash`
- 新的 `active` 版本导入后，应自动把同 `knowledge_key` 的旧 `active` 文档整体降级为 `deprecated`
- 默认检索只查 `active` 子文档，不把历史版本和当前版本混在一起召回

参考合同：

- `research/first-stage-rag-boundary.md`

## 9. 后续可继续展开的方向

一旦这份设计被确认，下一层规格可以继续细化：

- 场景级样例
- 子意图 seed list
- 回复结构要求
- 审核与升级规则
- 实现切片方式

当前已补充的第一阶段分支与协同合同：

- `research/first-stage-pre-sales-branch.md`
- `research/first-stage-after-sales-branch.md`
- `research/first-stage-business-data-rag-coordination.md`
- `research/first-stage-review-and-fallback.md`
- `research/first-stage-reply-state-model.md`
- `research/first-stage-observability-replay.md`

其中应特别坚持：

- 售前推荐优先给 2 到 3 个候选方向，而不是单点拍板
- 售后回复优先输出“当前事实 + 下一步动作 + 预期路径”
- 同一封邮件若同时依赖业务数据与 RAG，必须先 facts、后 knowledge
- 审核与兜底层必须能区分放行、修改、人工审核、阻塞
- 回复状态至少要显式区分草稿就绪、先追问、待人工审核、阻塞
- 每个关键节点都要保留最小结构化事件，支持按 `message_id` 或 `run_id` 回放
