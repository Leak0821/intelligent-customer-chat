# 设计：邮件 Agent 最小闭环

## 1. 设计目标

第一版要表现得像一个真实客服系统的主干，而不是一条单点演示路径。

这意味着：

- 有共享接入主链路
- 有显式的归一化处理层
- 有按场景分支的处理方式
- 有外部业务数据参与的回复生成
- 从一开始就保留兜底与审核节点

## 2. 采用的结构

本设计采用一条公共主链路 `C`：

```text
邮件接入
  -> 邮件清洗
  -> 意图重写
  -> 一级场景 + 子意图映射
  -> 上下文与画像加载
  -> 进入售前或售后分支
  -> 业务数据查询与/或 RAG 补充
  -> 生成回复草稿
  -> 校验 / 审核 / 发送前状态
```

## 3. 为什么采用这种结构

### 3.1 比单点 Demo 更稳

单点 demo 虽然更容易完成，但在以下场景很容易崩塌：

- 新增意图
- 引入售后支持
- 引入更复杂的业务数据查询

采用公共主链路，可以避免后续每加一个能力就重写一遍系统骨架。

### 3.2 更适合面试展示

这套结构能体现：

- 领域拆解能力
- 分阶段决策能力
- 真实业务数据优先原则
- RAG 只是增强，不是真相源
- 通过审核和兜底体现上线可信度

## 4. 关键层决策

### 4.1 意图重写层必须显式存在

原始邮件往往：

- 噪声很大
- 同时包含多个问题
- 信息不完整
- 情绪化表达较强

因此“意图重写 / 归一化”不应被隐藏成实现细节，而应被视为正式一层。

这样后续才能继续明确：

- 什么内容需要被重写
- 什么内容不能被改写
- 重写质量如何验证

### 4.2 先一级场景，再子意图

设计上先判断：

- 是售前？
- 还是售后？

然后再映射到更细的子意图。

这样可以保证分支结构稳定，同时允许意图体系逐步扩展。

### 4.3 先加载上下文，再做重推理

系统在进入更深层判断前，应先加载可用上下文。

原因是：一旦忽略历史信息，用户体验会迅速下降。

尤其在以下场景更明显：

- 售后多轮追问
- 老客户重复来信
- 售前推荐链路中的连续追问

### 4.4 业务事实优先于自由生成

当订单或物流事实存在时，回复应先围绕事实展开。

这对售后尤其关键，因为“礼貌但模糊”的回答，往往比“简洁但明确”的事实回复更糟。

### 4.5 审核节点是主链路的一部分

审核或校验节点不是以后再加的外挂，而是第一版就应保留的结构。

原因包括：

- 最终发送策略尚未定稿
- 高风险回复需要安全检查点
- 面试和真实落地都更看重系统可信度

## 5. 第一版取舍

### 已接受

- 邮件作为第一阶段唯一渠道
- 只覆盖少量高频子意图
- 画像信号在不可用时可以降级
- 回复草稿作为第一阶段稳定输出物

### 延后处理

- 完整意图树
- 最终发送模式
- 所有高级异常流程
- 广义投诉与补偿处理
- 多渠道编排

## 6. 需要警惕的风险

- 子意图范围过早膨胀
- 如果没有真实样本，意图重写很容易过拟合
- 画像数据在早期可能缺失或质量不稳定
- 如果没有坚持业务数据优先，RAG 会被误用
- 多问题邮件会暴露主问题 / 附带问题的处理缺口

## 7. 后续可继续展开的方向

一旦这份设计被确认，下一层规格可以继续细化：

- 场景级样例
- 子意图 seed list
- 回复结构要求
- 审核与升级规则
- 实现切片方式
