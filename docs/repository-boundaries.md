# 仓库入库边界

## 1. 原则

仓库只保存三类内容：

- 团队需要共享的
- 后续需要复盘的
- 可以被审查和复用的

凡是只对某一台机器、某一个账号、某一次安装过程有效的内容，默认留在本地。

## 2. 当前仓库建议入库的内容

| 类型 | 例子 | 处理建议 |
|---|---|---|
| 项目说明 | `README.md`、`CHANGELOG.md`、`AGENTS.md` | 入库 |
| 规范文档 | `docs/**` | 入库 |
| 探索记录 | `research/**` | 入库，但明确其不是最终规格 |
| 正式规格 | `openspec/**` | 入库 |
| 可复用脚本 | `scripts/**` | 入库，前提是不含密钥和个人路径依赖 |
| 样例模板 | `*.example`、脱敏示例配置 | 入库 |

## 3. 当前仓库必须留本地的内容

| 路径或类型 | 原因 | 处理建议 |
|---|---|---|
| `.local-ssh/` | 私钥、公钥、known_hosts 属于敏感材料 | 不入库 |
| `.codex-home/` | 本地运行态、插件启用状态、绝对路径配置 | 不入库 |
| `.codex-local/` | 第三方工具克隆仓库、本地 marketplace、副产物 | 不入库 |
| `.DS_Store` | 系统生成文件 | 不入库 |
| 本地 token / cookie / 浏览器状态 | 账户敏感信息 | 不入库 |
| 临时日志、缓存、导出物 | 不稳定且不可复用 | 不入库 |

## 4. 对当前已存在文件的判断

| 路径 | 判断 | 说明 |
|---|---|---|
| `README.md` | 可入库 | 仓库入口文档 |
| `CHANGELOG.md` | 可入库 | 变更记录 |
| `scripts/use-local-codex.sh` | 可入库 | 启用本仓库本地 Codex 环境的脚本 |
| `.codex-home/config.toml` | 留本地 | 含本机绝对路径和本地插件状态 |
| `.codex-home/**` | 留本地 | 运行态目录 |
| `.codex-local/**` | 留本地 | 下载或克隆的第三方工具与市场目录 |
| `.local-ssh/**` | 留本地 | 密钥材料 |
| `跨境电商AI邮件智能客服【Java小白终极落地细节手册】（导图100%全覆盖补全版）.pdf` | 默认留本地 | 大体量参考资料，且是否适合分发需单独确认 |

## 5. 如果要共享插件环境，正确做法是什么

不要把本地安装后的插件目录直接提交到仓库。正确做法是共享下面这些内容：

1. 安装说明
2. 安装脚本
3. 所需插件清单
4. 脱敏后的配置示例

也就是说，共享“如何搭起来”，而不是共享“我机器上已经搭好的运行态目录”。

## 6. 以后新增文件时的判断标准

提交前自检：

1. 这个文件换一台机器还成立吗
2. 这个文件不给团队也不会影响协作吗
3. 这个文件是否包含密钥、绝对路径、账号信息、缓存或生成物

如果第 1 条回答是否定，或第 3 条回答是肯定，默认不入库。
