# 智能客服项目 Agent 最佳实践 V1

## 1. 目的

这份文档用于初始化本仓库的 AI 协作方式，先统一文档结构、变更流程和落地边界，再逐步补技术栈细节。

当前版本只解决三件事：

- 文档怎么分层
- 需求怎么从探索进入实现
- 哪些文件应该入库，哪些必须留本地

当前版本刻意不锁定：

- 编程语言和框架
- 模型厂商与编排框架
- 向量库、缓存、消息队列等基础设施

## 2. 核心原则

### 2.1 先想清楚，再写代码

先明确目标、成功标准、非目标和验证方式，再进入实现。需求不清时先做研究，不直接写业务代码。

### 2.2 文档先行，规格单点生效

探索阶段的材料可以发散，但一旦进入实现，`openspec/changes/` 里的当前变更文档就是唯一合同源。

### 2.3 只做当前问题需要的最小方案

仓库现在还在项目初始化期，不做“为未来铺路”的抽象，不提前绑定技术路线，不顺手扩 scope。

### 2.4 结果必须可验证

每次变更都要留下证据，至少包括改动文件、执行命令、测试结果和已知风险。

### 2.5 可共享的入库，不可共享的留本地

仓库保存可复用、可审查、可协作的内容。本地运行态、私钥、缓存、机器路径、第三方工具克隆副本不进版本库。

## 3. 采用的文档组织思路

结合常见生产实践，这个仓库先采用 `docs-as-code + 分层文档` 的组织方式：

- `README.md`
  仓库入口，说明项目目标、当前状态和主要文档入口。
- `AGENTS.md`
  Agent 执行合同，约束默认行为、优先级和交付要求。
- `docs/`
  放团队级约定、方法说明、边界说明、设计原则。
- `research/`
  放探索记录、对比分析、问题澄清、未定事项。
- `openspec/changes/`
  放进行中的正式变更合同。
- `openspec/specs/`
  放已经稳定、当前生效的系统规格。

这样做的目的不是把文档变多，而是把“探索材料”和“实现依据”分开，避免后面边讨论边改代码。

## 5. 文档格式约定

### 5.1 一页只解决一个问题

每份文档应尽量保持单一目标，不要把背景、流程、操作步骤、长期规格、排障说明混在一页里。

### 5.2 开头先给读者定位

推荐每份核心文档开头先写清楚：

- 这份文档解决什么问题
- 适用于谁
- 不覆盖什么
- 读完之后应该得到什么

### 5.3 标题尽量任务化、可扫描

优先使用任务导向或判断导向标题，例如：

- `如何创建新的变更规格`
- `哪些文件可以提交到仓库`
- `什么时候需要转人工`

避免纯内部视角标题，例如：

- `关于变更的一些说明`
- `插件目录介绍`

### 5.4 顺序步骤用编号，枚举信息用列表

有先后顺序的内容用编号；只是并列说明时用无序列表；需要横向比较时再用表格。

### 5.5 少写长段落，多写结构化信息

对于流程、职责、输入输出、边界、风险，优先使用列表或表格，降低扫读成本。

## 6. 默认工作流

### 阶段 1：需求进入

先明确：

- 这次要解决什么问题
- 谁会使用这项能力
- 成功标准是什么
- 这轮明确不做什么

### 阶段 2：探索与澄清

遇到以下情况先写 `research/`：

- 需求模糊
- 有多种可行路径
- 涉及客服话术、升级路径、知识库边界、安全策略
- 技术选型尚未决定

这一阶段允许发散，但不直接修改业务实现。

### 阶段 3：形成正式变更合同

当需求准备进入实现时，在 `openspec/changes/<change-id>/` 下创建：

- `proposal.md`
- `spec.md`
- `tasks.md`
- `design.md`，仅在架构、数据流、接口、兼容性或安全复杂度较高时需要

约束：

- `proposal.md` 写背景、目标、非目标
- `spec.md` 写行为约束和验收标准
- `tasks.md` 拆成可验证的小任务
- 只有确认后的结论才能进入 OpenSpec

### 阶段 4：实现

实现时遵守：

- 一次只做一个任务
- 代码改动以当前 OpenSpec 为边界
- 不顺手重构
- 不把探索阶段的候选想法偷偷实现进去

### 阶段 5：验证

验证至少覆盖：

- 功能是否符合规格
- 是否有明显回归
- 是否留下可复盘证据

如果后续涉及接口、RAG、工作流引擎或人工转接，再补分层测试和评测规范。

### 阶段 6：归档

变更完成并验证后，把已经稳定的行为沉淀到 `openspec/specs/`，避免“最终设计只存在历史对话里”。

## 7. 智能客服场景下，规格里必须回答的问题

下面这些问题，即使技术栈未定，也应在后续规格里逐步补齐：

| 维度 | 至少回答什么 |
|---|---|
| 用户对象 | 是终端用户、客服坐席、运营人员，还是管理员 |
| 触达渠道 | 邮件、Web 聊天、站内 IM、WhatsApp，还是多渠道统一入口 |
| 触发条件 | 用户输入什么、系统事件是什么、上下文从哪里来 |
| 知识来源 | FAQ、商品规则、订单数据、工单系统、外部 API，还是人工编写知识 |
| 权限与边界 | 哪些问题可以直接答，哪些问题必须拒答或转人工 |
| 升级策略 | 什么情况下转人工、建工单、发通知或要求二次确认 |
| 观测与评估 | 如何看日志、如何回放、如何评估效果、如何发现幻觉或越权回复 |

## 8. 当前推荐的文档节奏

在正式进入功能开发前，优先按这个节奏推进：

1. 先稳定仓库级协作规则
2. 再沉淀一版客服场景范围定义
3. 再选第一条最小闭环链路做 OpenSpec
4. 最后才开始落技术方案和代码

对当前仓库来说，这意味着先把规范跑通，比马上选框架更重要。

## 9. 当前未决但需要后续确认的事项

后续文档迭代时，建议优先确认以下问题：

- 第一阶段到底做哪条客服链路
- 是否先只做单渠道
- 是否需要知识库检索
- 是否需要人工转接
- 是否需要多租户或账号体系
- 是否需要历史会话记忆
- 评测指标先看哪些

这些问题不在本版一次性定死，后面按 OpenSpec 逐步固化即可。
