# 项目范围定义

## 1. 目的

这份文档用于定义本项目当前阶段的业务边界，帮助后续 brainstorming、OpenSpec 和实现工作围绕同一条主线展开。

当前目标不是确定技术方案，而是先明确：

- 当前阶段解决什么问题
- 当前阶段先做哪些场景
- 当前阶段明确不做哪些事情
- 什么标准算这一阶段跑通

## 2. 项目定位

本项目要打造的是一个面向跨境电商邮件场景的智能客服 agent。

触发方式：

- 海外客户通过邮件发起咨询
- 系统读取邮件内容和上下文
- 智能体结合历史邮件、用户画像、订单信息、知识库信息进行理解和处理
- 生成建议回复，后续再决定是否人工审核后发送或自动发送

当前文档统一使用“产品”这一去敏表述，不在规范文档中写具体品类名称。

## 3. 当前阶段目标

当前阶段的目标是先跑通“跨境电商邮件智能客服”的最小业务闭环，而不是一次性做完整客服平台。

最小闭环定义为：

1. 接收客户邮件
2. 理解客户意图
3. 结合上下文与外部信息源完成处理
4. 生成可用的回复邮件草稿

这里的重点是“能正确处理典型邮件咨询并给出可用回复”，不是先把所有渠道、所有意图、所有自动化能力一次做完。

## 4. 当前阶段范围内

### 4.1 渠道范围

- 只聚焦邮件客服场景
- 输入主体是客户来信
- 输出主体是回复邮件草稿

### 4.2 核心能力范围

- 邮件内容理解
- 意图识别
- 上下文管理
- 历史邮件加载
- 用户画像补充
- 外部信息查询
- 基于知识库的补充回答
- 回复内容生成
- 回复前校验或评审节点

### 4.3 外部信息源范围

当前先按下面几类信息源设计：

- 历史邮件上下文
- 用户历史订单信息
- 浏览记录或购物车信息
- 订单系统查询能力
- 物流或履约相关信息查询能力
- RAG 知识检索能力

这些能力后续可以通过 MCP、skill、CLI 或其他方式接入，但当前文档不锁定接入实现形式。

## 5. 当前阶段的一级业务场景

当前阶段至少覆盖两个一级业务场景：

- 售前咨询
- 售后咨询

注意：

- 这两个是一级场景，不是完整意图列表
- 每个一级场景下面后续还会继续细分多个意图
- 当前阶段先不冻结完整意图树，只先固定一级边界

### 5.1 售前咨询

目标：

- 帮助客户基于需求描述咨询合适产品

典型问题：

- 按规格或需求推荐适合的产品
- 产品差异、基础功能、适用场景、搭配建议
- 发货、库存、基础购买前咨询

当前阶段希望具备的处理能力：

- 识别客户需求关键词
- 结合产品知识、用户画像、历史行为给出匹配建议
- 在不确定时给出保守回复，而不是编造信息
- 生成结构清晰的英文邮件回复草稿
- 支持客户二次追问后的连续推荐或对比说明

### 5.2 售后咨询

目标：

- 基于订单相关信息协助处理客户售后问题

典型问题：

- 查询订单状态
- 查询物流进度
- 解释售后处理状态
- 针对常见问题给出标准化回复

当前阶段希望具备的处理能力：

- 从邮件中识别订单号等关键实体
- 查询订单和物流信息
- 结合历史邮件判断上下文
- 必要时结合售后规则或政策知识生成解释
- 对标准售后问题生成回复草稿

## 6. 当前阶段明确不做

为避免范围失控，当前阶段先不纳入以下内容：

- 多渠道统一接入
- 即时聊天或站内 IM
- 复杂工单系统
- 多租户能力
- 复杂账号权限体系
- 全量售后流程编排
- 高风险场景的全自动决策
- 所有细分意图一次性覆盖

下面这些点也暂时不定：

- 回复邮件是否自动发送
- 是否必须人工审核后发送
- 意图分类体系要细到什么程度
- 外部能力具体采用 MCP 还是 skill/CLI 接入
- 用户画像优先使用哪些数据源

这些问题后续可以在 brainstorming 或 OpenSpec 阶段再收敛。

## 7. 当前阶段建议能力边界

从 agent 视角，当前阶段先聚焦以下处理链路：

1. 收到邮件
2. 读取当前邮件正文
3. 关联客户历史邮件
4. 尝试补充用户画像信息
5. 识别一级场景和更细粒度意图
6. 提取关键实体
7. 调用对应信息源
8. 用 RAG 补充通用知识
9. 生成回复草稿
10. 进入评审、校验或发送前节点

其中关键实体至少包括：

- 客户问题类型
- 产品需求关键词
- 订单号
- 物流相关标识

如果可用，辅助上下文还可以包括：

- 历史购买记录
- 浏览记录
- 购物车信息

## 8. 当前阶段的回答与兜底原则

当前阶段默认遵守以下原则：

- 真实业务数据优先于模型自由生成
- RAG 用于补充知识，不用于替代真实订单或物流数据
- 信息不足时可以追问、保守回复或转人工
- 数据冲突、关键参数缺失、低置信度时不能强行生成确定答案
- 高风险场景优先保留人工审核或人工接管能力

## 9. 当前阶段成功标准

这一阶段完成，不意味着系统完全上线，而意味着下面这些能力已经形成稳定最小闭环：

1. 至少覆盖售前和售后两类主场景
2. 每个主场景下至少能继续承载多个细分意图，而不是只做单一问答模板
3. 能处理邮件上下文而不是只看单封邮件
4. 能按需查询订单或物流信息
5. 能在可用时利用用户画像补充判断
6. 能结合知识库降低明显幻觉
7. 能生成结构可用的回复邮件草稿
8. 能保留评审、校验或人工接管的扩展空间
9. 能沉淀后续继续优化所需的规格和验证基线

## 10. 当前阶段的关键约束

- 技术栈暂不锁定
- 工具接入方式暂不锁定
- 自动发送策略暂不锁定
- 文档和流程先行
- 范围优先收敛，不追求一次做大

## 11. 后续需要确认但暂不在本轮定死的问题

- 第一阶段邮件语言是否只聚焦英文
- 售前与售后的意图细分颗粒度
- RAG 的知识来源边界
- 历史邮件回溯的窗口范围
- 订单系统和物流系统的查询权限边界
- 浏览记录、购物车、用户画像是否稳定可取
- 回复草稿的语气、模板化程度和品牌风格
- 人工审核与自动发送的触发规则

## 12. 与后续文档的关系

- 这份文档定义“当前阶段做什么、不做什么”
- `docs/ai-workflow.md` 定义“如何协作推进”
- `openspec/changes/` 负责定义“某次具体变更如何落地”

也就是说，这份文档是后续 brainstorming 和 OpenSpec 的业务边界底稿。
