# 开发踩坑记录

这份文档用于持续记录项目开发过程中已经真实遇到过的问题、现象、根因和修复方式。

目的不是写成流水账，而是把后续最容易重复踩的坑提前说明白。

## 1. 上下文压缩窗口被截断后，不再触发重压缩

### 现象

- 线程消息达到压缩阈值后，会生成一次摘要
- 但后续新邮件继续进入线程时，摘要不再更新

### 根因

- 早期实现用“最近窗口消息数”判断摘要是否已覆盖
- Redis 里最近消息列表本身会被裁剪到固定长度
- 一旦窗口大小固定，后续即使有新消息进来，列表长度也不再变化
- 导致系统误以为“摘要已经覆盖到最新状态”

### 修复

- 为线程单独维护累计消息数
- 判断摘要是否需要重算时，对比累计消息数而不是窗口长度

### 经验

- 任何“基于滑动窗口的数据”都不能直接当版本号使用
- 只要后面存在裁剪、压缩、淘汰，就必须有独立的累计计数或版本号

## 2. Redis 中最近消息顺序与真实会话顺序相反

### 现象

- 摘要内容读起来顺序混乱
- 最新邮件会排在最前面，导致会话演进不自然

### 根因

- Redis 列表采用头插
- 摘要服务直接拿列表内容送去压缩，没有先反转顺序

### 修复

- 压缩前先把最近消息恢复成时间正序

### 经验

- 缓存结构的存储顺序，不一定等于业务语义顺序
- 进入 LLM 前，要先确保输入顺序符合人类阅读逻辑

## 3. Spring AI API 版本细节不能靠印象写

### 现象

- 写 `ChatClient.Builder`、`ChatOptions` 或 embedding 调用时，容易出现方法签名不匹配

### 根因

- Spring AI 版本迭代快
- 不同版本里的 builder、options、model 接口细节会变化

### 修复

- 直接对本地 `.m2` 中的依赖包做 `javap`
- 以当前仓库真实依赖版本为准写代码，不靠记忆

### 经验

- 只要是新库或更新快的库，先查本地真实 class 签名
- 这比查二手博客可靠得多

## 4. LLM 会尝试放松关键实体约束

### 现象

- 售后邮件缺少订单号或物流号时，模型仍可能给出“可以继续处理”的结构化输出

### 根因

- 模型擅长补全“看起来合理”的结构
- 但关键实体是否真的存在，不能只靠模型判断

### 修复

- 保留启发式护栏层
- 对订单号、物流号等关键实体做规则校验
- 即使模型输出 `CONTINUE`，规则层仍可强制回退到 `FOLLOW_UP`

### 经验

- LLM 适合做归一化和重写，不适合单独承担真值确认
- 关键实体必须有规则层或业务层硬校验

## 5. 订单号 / 物流号正则很容易误判

### 现象

- 有时把普通英文单词误识别成编号
- 有时又识别不到常见表达，比如 `order number is AB12345678`

### 根因

- 正则一开始过于宽松或过于死板
- 真实邮件的表达方式比想象中更散

### 修复

- 收紧编号主体规则，避免把普通单词当编号
- 同时补充 `number is XXX`、`tracking id XXX` 之类常见表达
- 用单测锁住正反两类场景

### 经验

- 这类正则不要只做“识别成功”测试
- 必须同时做“不能误识别”的反向测试

## 6. demo 直调流程时，没有 mail receipt 会影响发送链路

### 现象

- 通过 demo 接口直接 `process(inboundMail)` 跑出来的工作流可以生成草稿
- 但进入发送链路时，会因为查不到 `mail_receipt` 而失败

### 根因

- 早期 `mail_receipt` 主要服务于 IMAP 拉取链路
- demo/manual 入口并不会自动补一条 receipt
- 发件链路拿收件人时又依赖这条 receipt

### 修复

- 对非 IMAP 的手动入口，补一条合成 `mail_receipt`
- 让发送链路不再只适用于某一种接入来源

### 经验

- 只要有多种入口，就要警惕某条链路只对“主入口”有效
- demo 入口也要尽量遵守正式数据约束，否则后面展示时最容易暴露问题

## 7. 静态知识检索如果只返回假文案，演示价值很低

### 现象

- 早期静态知识检索只是返回一条格式化字符串
- 虽然接口不报错，但 `analysis` 看起来像空壳

### 根因

- 过度依赖“以后接 ES 再说”
- 没有给非 ES 环境准备最小可用的知识样本

### 修复

- 新增内置知识样本目录
- 静态检索改成基于真实样本切块和打分
- 不开 ES 也能返回像样的知识证据

### 经验

- 所有外部依赖都应该有“非生产但可演示”的降级形态
- 否则一旦现场环境不齐，演示效果会直接坍塌

## 8. “草稿已生成”不等于“已经可以发送”

### 现象

- 如果只有 `draft_ready / follow_up_needed / human_review_required / blocked`
- 后续很难表达“草稿存在，但仍等待审核”或“审核后可发送”

### 根因

- 早期把“草稿状态”和“发送前状态”混在了一起

### 修复

- 为 `ReplyDraft` 增加 `send_readiness` 与 `next_action`
- 把“生成草稿”和“允许发送”拆开
- 再通过 `approve-send` 和 `dispatch` 显式推进

### 经验

- 真实系统里，一个对象经常同时有“内容状态”和“流程状态”
- 这两个维度不要强行合并

## 9. 发送适配层第一版不要直接绑死真实 SMTP

### 现象

- 如果一开始就强依赖真实 SMTP 或第三方发信服务，开发和测试成本会立刻变高

### 根因

- 发信链路天然带副作用
- 一旦配置错误，还可能误发真实邮件

### 修复

- 第一版先落 `OutboundMailSender` 抽象
- 默认接 `NoopOutboundMailSender`
- 测试和 demo 走完整发送链路，但不做真实外发

### 经验

- 对所有有副作用的能力，第一版都应该优先提供安全的 no-op 或 sandbox 适配器

## 10. 多个 SpringBootTest 上下文复用同名 H2 内存库，会出现重复建表

### 现象

- 单个测试类独立运行正常
- 但新增一个带自定义 Bean 覆盖的 `SpringBootTest` 后，整套测试跑起来会在 `schema.sql` 初始化时报 `table already exists`

### 根因

- H2 内存库名称相同，且配置了 `DB_CLOSE_DELAY=-1`
- 前一个测试上下文关闭后，内存库不会立刻销毁
- 新上下文再次执行 `schema.sql` 时，就会撞上已有表结构

### 修复

- 对需要独立上下文的测试，显式改用新的 H2 内存库名
- 不要默认假设“测试类变多了也一定共用同一上下文”

### 经验

- 只要测试引入 `@MockBean`、`@MockitoBean`、额外属性覆盖或其他上下文差异，就可能触发新的 Spring 上下文
- 这时 H2 库名最好也跟着隔离，否则最容易出现初始化脚本互相污染

## 11. 如果直接绕过审核服务，只调发送放行服务，会丢失审核审计信息

### 现象

- 草稿能正常从“待审核”推进到“可发送”
- 发送链路也能继续执行
- 但回放里看不到明确的审核人和审核意见

### 根因

- `ReplySendLifecycleService` 只负责发送状态推进
- 如果直接调用它的放行方法，而不经过专门的审核生命周期服务
- 就不会落 `review_records`

### 修复

- 把管理接口的批准/驳回动作统一收敛到审核生命周期服务
- 审核服务负责同时做两件事：
  1. 推进草稿审核状态
  2. 落审核留痕

### 经验

- 审核动作和发送动作虽然相邻，但不是一回事
- 只要系统需要“谁批准了什么”，审核动作就必须有独立的审计出口

## 12. 业务查询如果长期停留在硬编码桩数据，后面的售后演示会越来越假

### 现象

- 工作流能跑通，但订单状态、物流节点、售后策略始终是固定文案
- 一旦开始做人工审核、重试补偿、知识检索联动，演示内容会显得很空

### 根因

- 早期为了快速打通主链路，业务查询网关直接返回硬编码结果
- 这种方式适合切片 1，不适合继续承接后续切片

### 修复

- 新增本地业务数据目录 `LocalBusinessDataCatalog`
- 让订单、物流、售后策略网关统一从目录取数
- 再补管理接口，支持运行时查看和更新样本数据

### 经验

- 桩数据不是不能用，而是要尽快从“写死在代码里”升级到“可维护的本地数据源”
- 这样后续替换真实系统时，应用层编排才能保持稳定

## 13. 草稿版本不能和审核状态变更混为一谈

### 现象

- 刚生成的一版草稿，看起来还没被人工改过
- 但版本号却已经变成 `2` 或更大

### 根因

- 早期把“审核结果写回草稿状态”和“人工真正修改草稿内容”共用了同一个 `revise`
- 导致只要经过自动审核节点，版本号就会增长

### 修复

- 把“内容修订”和“审核结果回写”拆成两个动作
- 只有真正修改 `subject/body` 时才递增 `draft_version`
- 纯审核状态变化只更新状态和备注，不增加版本

### 经验

- 版本号要描述“内容演进”，不要描述“流程经过了几个节点”
- 否则后面做人工审核台时，版本语义会很快失真

## 14. 后续记录原则

后续继续记录问题时，建议按这个格式补：

1. 现象
2. 根因
3. 修复
4. 经验

不要只写“修好了”，要写清楚为什么会出问题，以及下次怎么避免。
