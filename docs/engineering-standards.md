# 工程实现原则

## 1. 目的

这份文档提炼与技术栈无关的实现原则，作为后续真正开始写代码时的上位约束。

它不规定具体框架写法，不绑定 Java、Node.js、Python 或任何单一技术路线。

## 2. 顶层原则

### 2.1 Think Before Coding

先明确目标、边界、风险、成功标准和验证方法，再开始实现。

禁止一边猜需求一边写代码。

### 2.2 Simplicity First

优先选择能满足当前目标的最小方案。

禁止为了假设中的未来需求提前做抽象、分层或平台化设计。

### 2.3 Surgical Changes

改动必须与当前任务直接相关。

禁止顺手重构、顺手统一风格、顺手清理无关历史问题。

### 2.4 Goal-Driven Execution

每次任务都要有明确完成标准，并留下可验证证据。

没有验证证据，不算完成。

## 3. 开始实现前必须明确的 5 件事

1. 这次改动解决什么问题
2. 成功标准是什么
3. 明确不做什么
4. 最小改动面在哪里
5. 用什么证据证明完成

## 4. 编码阶段通用规则

### 4.1 不硬编码敏感信息

禁止把密钥、token、密码、私有地址和账号凭证直接写入代码、脚本或文档示例。

### 4.2 不绕过分层边界

无论后续选什么技术栈，都要避免把职责混在一起。

至少要保证：

- 输入处理
- 业务决策
- 数据访问
- 外部系统调用

这些职责不要无边界地堆在同一层。

### 4.3 不为了“看起来通用”而过度抽象

如果一个抽象还没有被当前真实场景证明需要，先不要引入。

### 4.4 修改必须能回到任务目标

每个主要改动都应该能回答一句话：

“它为什么对当前任务是必要的？”

答不上来，就不要改。

## 5. 验证原则

### 5.1 验证必须贴合改动风险

小改动做定向验证，大改动做完整验证。

不要完全不测，也不要用与风险不匹配的超重流程拖慢节奏。

### 5.2 验证证据要可复盘

优先保留：

- 命令
- 测试结果
- 日志结论
- 核心输入输出

### 5.3 先区分问题类型，再修

验证失败时，先判断属于哪一类：

- 当前实现 bug
- 测试问题
- 规格缺口
- 既有无关问题
- 环境问题

只修和当前变更直接相关的问题。

## 6. 代码审查原则

后续进入 PR 或 diff 审查时，优先关注：

1. 是否偏离任务目标
2. 是否扩大了范围
3. 是否缺少验证证据
4. 是否引入安全或数据风险
5. 是否存在明显不可维护的复杂度

## 7. 适用于当前仓库的特别说明

当前项目仍在规范建设期，因此这些原则先作为“准备进入编码前”的门槛使用：

- 还没定技术栈时，不提前写框架级细则
- 还没定第一条链路时，不提前写业务实现规范
- 先沉淀流程和边界，再补代码模板和项目级约束
