# 第一阶段 Docker Compose 基础设施方案

## 1. 目的

这份文档用于定义第一阶段本地开发和联调时的 Docker 基础设施组合。

目标：

- 不污染宿主机本地环境
- 让依赖组件可重复拉起
- 为后续编码和联调提供统一基础环境

## 2. 第一阶段建议容器清单

第一阶段建议放进 `docker-compose.yml` 的组件：

1. `mysql`
2. `redis`
3. `elasticsearch`
4. `nacos`
5. `xxl-job-admin`

应用本身建议：

- 继续本地 IDE 启动
- 不急着第一阶段就把应用也容器化

原因：

- 本地断点调试更方便
- 先把应用开发效率放在第一位

## 3. 每个组件的用途

### 3.1 MySQL

用途：

- 工作流运行记录
- 回复草稿
- 长期用户档案
- 审核记录

建议：

- 独立持久化 volume
- 初始化脚本与建表脚本后续单独挂载

### 3.2 Redis

用途：

- 短时上下文缓存
- 最近会话摘要
- 幂等辅助键
- 临时状态缓存

建议：

- 第一阶段先用单实例
- 不做哨兵或集群

### 3.3 Elasticsearch

用途：

- 知识库混合检索
- 向量检索
- BM25 检索
- RRF 融合前的数据承载

建议：

- 第一阶段采用单节点
- 使用安全配置最小化的开发模式
- 保留 data volume，避免每次重建都重索引

### 3.4 Nacos

用途：

- Prompt 配置
- 意图枚举配置
- 检索参数
- 审核阈值
- 非敏感运行参数

建议：

- 第一阶段先用单机模式
- 后续再考虑 namespace、group、profile 的更细分规划

### 3.5 XXL-JOB Admin

用途：

- IMAP 拉邮调度
- 索引更新调度
- 上下文压缩调度
- 重试和补偿调度

建议：

- 第一阶段先起 admin
- executor 在业务应用里内嵌

## 4. 网络与端口建议

建议：

- 所有基础设施放在同一个 compose network
- 端口映射尽量固定，方便本地配置

示例方向：

- MySQL：`3306`
- Redis：`6379`
- Elasticsearch：`9200`
- Nacos：`8848`
- XXL-JOB Admin：`8080` 或其他本地不冲突端口

最终端口落地时，以避免和宿主机现有端口冲突为准。

## 5. 配置与密钥处理建议

第一阶段建议：

- `docker-compose.yml` 只保留必要环境变量占位
- 敏感值通过 `.env` 或本地环境变量注入
- `.env` 不入库

不要做的事：

- 把模型 API Key 直接写进 compose 文件
- 把邮箱密码写进版本库

## 6. 第一阶段不建议一开始就做的事情

第一阶段不建议：

- 一开始就做多节点 ES
- 一开始就做 Redis Cluster
- 一开始就做 Nacos 集群
- 一开始就把所有组件 Kubernetes 化

当前重点不是“像生产一样复杂”，而是“像生产一样可迁移、可复现、可解释”。

## 7. 当前建议结论

第一阶段本地环境应采用：

- 应用本地启动
- 基础设施通过 Docker Compose 承载

这是当前阶段最稳、开发体验最好、也最适合面试展示的组合。
