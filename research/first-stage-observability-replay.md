# 第一阶段日志、回放与评估最小要求

## 1. 目的

这份文档用于定义第一阶段最小闭环必须具备的可观测性和回放能力。

它的重点不是“大而全的监控平台”，而是保证：

- 链路出了问题能查
- 能证明这条闭环不是假跑
- 能为后续优化积累最小评估材料

## 2. 第一阶段至少要记录什么

第一阶段建议至少记录以下事件：

1. 邮件接入事件
2. 邮件清洗结果
3. 意图归一化结果
4. 场景与子意图识别结果
5. 上下文加载结果
6. 业务数据查询结果
7. RAG 检索结果
8. 回复规划与草稿生成结果
9. 审核或兜底决策结果

## 3. 每个事件建议的最小字段

建议字段：

- `message_id`
- `thread_id`
- `run_id`
- `stage`
- `status`
- `timestamp`
- `summary`

可选增强字段：

- `input_snapshot_ref`
- `output_snapshot_ref`
- `error_code`
- `latency_ms`

## 4. 第一阶段回放要能回答什么问题

一条完整回放至少应能回答：

- 这封邮件进来后被识别成什么场景和子意图
- 是否命中了上下文
- 是否调用了业务查询
- 是否调用了 RAG
- 最终是直接出草稿、追问还是转人工
- 如果被拦截，拦截原因是什么

## 5. 第一阶段不要求的内容

第一阶段不要求：

- 完整 APM 平台
- 复杂分布式 tracing 体系
- 全量 token 级观测平台
- 成熟 BI 评估后台

但不代表可以完全没有结构化事件链。

## 6. 第一阶段评估最小要求

第一阶段建议至少保留以下评估样本字段：

- 原始邮件摘要
- 归一化结果
- 一级场景与子意图判断
- 是否触发业务查询
- 是否触发 RAG
- 最终输出状态
- 人工备注或评审结论

这样后续至少能做：

- 错误案例复盘
- 子意图覆盖缺口分析
- 追问是否过多的初步判断
- 转人工原因统计

## 7. 第一阶段建议结论

第一阶段的“可观测性”最少也要做到：

- 每个关键阶段有结构化事件
- 能按 `run_id` 或 `message_id` 回放主链路
- 能看到最终为什么没有直接出草稿
- 能沉淀最小评估样本，而不是只能靠人工记忆复盘
