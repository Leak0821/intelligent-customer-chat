# 第一阶段技术设计决策清单

## 1. 目的

这份文档用于补足当前 `review-spec / design` 阶段还没有完全收敛的技术决策点。

它不直接锁死最终技术栈，而是明确：

- 第一阶段最小闭环实现前，哪些技术问题必须回答
- 每个问题有哪些常见路径
- 当前更推荐哪种方向
- 哪些问题可以继续后置

## 2. 使用方式

当前文档的角色是“技术决策清单”，不是最终实现合同。

后续流程建议：

1. 先用这份清单收敛关键技术点
2. 再把确认结论回写到当前 OpenSpec `design.md`
3. 然后进入 `gstack` 审设计
4. 审完再拆实现任务

## 3. 必须先收敛的技术决策

### 3.1 邮件接入方式

要回答的问题：

- 邮件是怎么进入系统的
- 第一阶段到底是主动拉取还是被动接收
- 失败重试和重复消费怎么控制

常见路径：

- IMAP / POP3 轮询拉取
- 邮件服务商 webhook / API 推送
- 第三方客服平台转发

当前建议：

- 第一阶段优先选“稳定、容易验证、容易回放”的接入方式
- 如果真实业务环境已经有固定邮件服务入口，应优先贴合现有环境
- 如果没有既有环境约束，优先选择便于调试和重复回放的方案

当前不必定死的内容：

- 最终是否支持多邮箱接入
- 是否接入完整客服平台

### 3.2 主流程编排方式

要回答的问题：

- 这条最小闭环由什么来组织
- 状态如何在每个节点之间传递
- 多步处理如何避免流程混乱

常见路径：

- 应用代码内顺序编排
- 显式状态机
- Agent workflow framework
- 异步任务队列 + 状态流转

当前建议：

- 第一阶段优先选择“可追踪、可调试、状态清楚”的方案
- 不建议为了面试或概念先进性过早引入过重框架
- 主链路状态要可落库或可回放，而不是只靠内存串起来
- 当前项目更推荐采用 `Java + Spring Boot` 作为实现底座
- 主流程建议采用“显式状态主链路”或“显式状态机”来组织，而不是先把主干交给黑盒 agent workflow framework
- LLM、Embedding、Tool Calling、RAG 等 AI 能力建议通过统一抽象层接入，避免业务流程直接依赖具体模型 SDK

当前不必定死的内容：

- 是否最终引入复杂 workflow framework
- 是否一开始就做分布式编排

当前推荐基线：

- 主应用：`Spring Boot`
- 主流程：`Java 显式状态主链路 / 状态机`
- AI 抽象层：优先采用 `Spring AI`
- 阿里模型接入补充层：按需引入 `Spring AI Alibaba`
- 生产级异步编排增强：后续可评估 `Temporal`，但不作为第一阶段默认前提

这样做的原因是：

- 对你当前项目，主流程可调试、可解释、可演示，比“框架看起来更先进”更重要
- `Spring AI` 适合作为主干抽象层，便于后续切换模型提供方
- `Spring AI Alibaba` 更适合作为阿里生态能力补充，而不是直接承载全部业务主流程
- 如果后面接入阿里云百炼、千问 embedding 或其他国内模型，这种边界会更稳

### 3.3 意图重写与意图匹配方式

要回答的问题：

- 意图重写和意图分类是一体还是分开
- 子意图枚举如何维护
- 多问题邮件的主问题如何识别

常见路径：

- 单次模型调用同时输出重写 + 一级场景 + 子意图
- 重写和分类分两步执行
- 规则前置 + 模型补足

当前建议：

- 第一阶段应优先追求稳定性和可解释性
- 比起“最省调用次数”，更重要的是结果可校验、可追踪
- 多问题邮件至少要能识别主问题和附带问题

当前不必定死的内容：

- 完整意图树
- 最终 prompt 细节
- 是否采用多模型协作

### 3.4 上下文与画像存储方式

要回答的问题：

- 短期会话放哪里
- 长期用户信息放哪里
- 画像信号如何分层
- 哪些是必需上下文，哪些是可选增强

常见路径：

- 单数据库统一存储
- 缓存 + 数据库分层
- 结构化档案 + 非结构化会话并存

当前建议：

- 第一阶段至少区分“当前会话上下文”和“长期业务档案”
- 历史邮件应视为必备上下文
- 浏览记录、购物车、用户画像可作为可选增强
- 第一阶段建议显式区分“必需上下文 / 强相关增强上下文 / 可选画像增强”
- 可选画像缺失不得阻塞主链路
- 上下文冲突时应坚持“业务真值优先、当前回合明确表达优先、旧画像最低优先级”

当前不必定死的内容：

- 最终缓存产品选择
- 长期记忆是否引入向量化用户画像

当前合同参考：

- `research/first-stage-context-loading-contract.md`

### 3.5 外部业务数据接入方式

要回答的问题：

- 订单系统怎么查
- 物流系统怎么查
- 是否通过 MCP、内部 API、skill 或 CLI 接入
- 是否需要中间适配层

常见路径：

- 直接调用内部 API
- 通过 MCP 统一接入
- 通过 skill / CLI 封装已有查询能力
- 做一层内部 adapter 统一屏蔽外部差异

当前建议：

- 第一阶段优先保留“接入能力抽象层”
- 不要让业务主流程直接绑死某一个接入方式
- 对面试和后续扩展来说，adapter 思路比具体协议更重要
- 第一阶段建议把订单、物流、标准售后规则能力统一收口到内部 gateway / adapter 边界
- 主流程不应直接感知下层到底是 MCP、内部 API、skill 还是 CLI
- 查询结果应结构化区分成功、条件不足、无结果、临时失败、结果冲突

当前不必定死的内容：

- 最终只用 MCP 还是混用 skill / CLI
- 所有系统是否一步接齐

当前合同参考：

- `research/first-stage-business-data-access-contract.md`

### 3.6 RAG 第一版落地方式

要回答的问题：

- 第一阶段是否上线 RAG
- 知识源先放哪些文档
- 检索返回内容怎么和业务数据协同

常见路径：

- 第一版先不上 RAG，只靠规则和业务数据
- 第一版上轻量 RAG，只覆盖产品和政策文档
- 第一版做完整知识库体系

当前建议：

- 第一阶段更适合“轻量 RAG”
- 只把产品知识、政策规则、常见 FAQ 作为首批知识源
- RAG 必须是补充层，不能覆盖订单/物流真值
- 第一阶段应限制知识源范围，优先选择来源明确、相对稳定、可解释的文档
- 检索结果应结构化输出，避免直接把大段拼接文本塞进主链路
- 回复阶段必须显式区分业务事实与知识补充

当前不必定死的内容：

- 最终向量存储产品
- 检索优化与重排序的复杂策略

补充建议：

- 如果后续使用阿里千问的 text embedding，建议把 embedding 能力放在统一 `EmbeddingService` 抽象后面
- 如果后续使用 `DeepSeek` 作为对话模型，优先通过 `Spring AI` 官方 provider 接入
- 如果后续接 `Qwen / DashScope`，更适合放在 `Spring AI Alibaba` 的 provider 适配层中
- 业务层不要直接感知“当前到底是 Spring AI 还是 Spring AI Alibaba 在提供底层模型能力”

当前合同参考：

- `research/first-stage-rag-boundary.md`

### 3.7 回复草稿、审核与发送前状态

要回答的问题：

- 回复草稿产物保存在哪里
- 审核节点如何表示
- 自动发送和人工审核如何切换

常见路径：

- 草稿写数据库，人工后台审核
- 草稿写邮件草稿箱
- 草稿进入工单或待处理队列

当前建议：

- 第一阶段先把“回复草稿状态”设计清楚
- 审核节点要可扩展，不要求当前就锁死最终发送模式
- 草稿、审核、发送前校验状态必须可追踪
- 第一阶段建议至少显式区分 `draft_ready / follow_up_needed / human_review_required / blocked`
- 发送前状态建议与草稿主状态分层建模，避免把“可发送”和“有草稿”混成一件事

当前不必定死的内容：

- 最终是否自动发送
- 最终审核界面长什么样

当前合同参考：

- `research/first-stage-review-and-fallback.md`
- `research/first-stage-reply-state-model.md`

### 3.8 日志、回放与评估

要回答的问题：

- 这条链路如何排障
- 之后如何评估回复质量
- 面试展示时如何证明闭环不是假跑

常见路径：

- 只记应用日志
- 记事件日志 + 节点状态
- 记 prompt / tool trace / 审核结果

当前建议：

- 第一阶段至少要有可回放事件链
- 至少能看到邮件输入、意图结果、工具调用结果、回复草稿、兜底原因
- 这部分不是锦上添花，而是“能否证明系统闭环可信”的关键
- 第一阶段建议每个关键节点都记录结构化事件，并能按 `run_id` 或 `message_id` 回放
- 第一阶段至少要能沉淀最小评估样本，而不是只保留零散日志

当前不必定死的内容：

- 最终评估平台
- 大规模观测体系

当前合同参考：

- `research/first-stage-observability-replay.md`

## 5. 当前已收敛的第一阶段技术基线

截至当前阶段，已经可以先固定以下基线：

- 邮件接入：第一阶段先采用 `IMAP`
- 主流程编排：采用 `Java + Spring Boot + 显式状态主链路`
- AI 能力主抽象：采用 `Spring AI`
- 阿里模型生态扩展：按需引入 `Spring AI Alibaba`

当前建议的模型接入边界：

- `DeepSeek`：优先走 `Spring AI` 官方接入
- `Qwen / DashScope Embedding`：优先放到 `Spring AI Alibaba` 适配层
- 业务层统一依赖自定义的 `LlmClient`、`EmbeddingService`、`ReplyPlanner` 等内部接口

当前确认说明：

- 第一阶段默认按 `Spring AI` 方向推进
- `Spring AI Alibaba` 仅作为后续接阿里模型生态时的补充接入层，不作为当前主流程基础

这份基线的目的不是提前锁死所有中间件，而是先保证第一阶段主干稳定：

- 主流程能解释
- 状态能追踪
- 模型可替换
- 后续接阿里生态时不需要推翻主链路
6. 回复草稿与审核前状态
7. RAG 第一版落地方式
8. 日志、回放与评估

## 5. 当前结论

当前阶段不需要一次锁死全项目技术栈，但必须先收敛“第一阶段最小闭环能否落地”所需的关键技术决策。

也就是说：

- 现在仍属于 OpenSpec 主导
- 可以对局部技术问题使用定向 brainstorming
- 但一旦形成结论，必须回写到 OpenSpec `design.md`

## 6. 当前推荐基线（优先级最高的 3 项）

这一节不是最终锁死方案，而是为后续 `gstack` 审设计提供一个“当前推荐基线”。

### 6.1 邮件接入方式：第一阶段先用 IMAP，保留接入适配层

当前已确认的第一阶段决策：

- 第一阶段先采用 `IMAP` 拉取邮件
- 在系统内部通过一层“邮件接入适配层”统一收口
- 后续如有需要，再扩展到事件推送或平台转发接入

为什么当前先选 IMAP：

- 成熟稳定
- 易于调试
- 易于回放
- 第一阶段接入不确定性更低
- 足够支撑最小闭环的真实落地

对当前项目的落地建议：

- 第一阶段先实现 `ImapMailSourceAdapter`
- 主流程通过统一接入层读取邮件，而不是直接依赖 IMAP 细节
- 已读状态、重复消费、失败重试和邮件去重应属于接入层职责

当前结论：

- 第一阶段邮件接入方式：`IMAP`
- 设计约束：业务主流程不得直接绑死在 IMAP 协议细节上

### 6.2 主流程编排方式：优先“显式状态主链路”，而不是过重框架先行

当前推荐：

- 第一阶段优先采用“显式状态主链路”
- 主链路节点和状态必须可追踪、可持久化、可回放
- 如果需要 agent workflow 框架，优先选择支持持久状态和人工介入的轻量方案

为什么这样推荐：

- 当前这条邮件 agent 主路径本质上更像 workflow，而不是完全自由探索的 agent
- LangGraph 官方文档明确区分了 workflow 和 agent：workflow 是预先定义好的代码路径，agent 才是动态决定工具和流程
- LangGraph 官方也强调了对长流程、持久执行、人工介入、短期和长期记忆、调试可见性的支持
- 这说明如果后续要上 agent workflow 框架，LangGraph 这类“状态明确、可追踪”的方案是合理候选

对当前项目的落地建议：

- 第一阶段主路径先按显式状态机思路设计
- 即使后续使用 LangGraph，也应先把主链路状态节点定义清楚，而不是先堆框架
- 面试场景下，清晰状态流比“用了更高级框架”更有说服力

不建议现在就做的事：

- 为了显得先进，过早引入过重的工作流平台
- 在主路径未稳定前就把系统做成过度动态的多 agent 流程

### 6.3 意图重写与意图匹配方式：优先“两步走 + 结构化输出”

当前推荐：

- 先做意图重写 / 归一化
- 再做一级场景与子意图匹配
- 两步之间优先使用结构化输出，而不是全靠自由文本串联

为什么这样推荐：

- 当前主链路已经明确存在噪声邮件、多问题邮件、关键信息缺失这些问题
- 如果把“重写”和“分类”完全揉成一个黑盒步骤，后续很难调试和校验
- LangGraph 官方 workflow 文档明确提到，构建 workflow 或 agent 时可以依赖支持 structured outputs 和 tool calling 的 chat model
- 对这个项目来说，结构化输出有助于：
  - 单独评估重写质量
  - 单独评估意图映射质量
  - 更稳地衔接工具调用与后续分支处理

对当前项目的落地建议：

- 第一阶段不要追求“最少一次模型调用”
- 更应该追求“每一步能看懂、能验证、能纠错”
- 多问题邮件里至少要显式产出主问题、附带问题或“信息不足需追问”的判断

不建议现在就做的事：

- 一上来就冻结完整意图树
- 过早优化成极致低调用成本而牺牲可解释性

## 7. 当前 review-spec 结论

如果按当前文档状态继续推进，下一步最合理的是：

1. 以这 3 个推荐基线作为当前设计草案
2. 把剩余技术点继续收敛成同样粒度的基线建议
3. 然后进入一次 `gstack` 设计审查
