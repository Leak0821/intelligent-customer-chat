# 第一阶段外部业务数据接入合同

## 1. 目的

这份文档用于明确第一阶段“订单、物流、售后等外部业务数据”如何接入系统主链路。

它要解决的问题不是“最终接哪个协议”，而是：

- 业务主流程应该依赖什么抽象
- 外部系统能力如何被统一收口
- 哪些查询是第一阶段必需
- 查询失败、无结果、数据冲突时如何处理

## 2. 设计原则

第一阶段应坚持以下原则：

- 业务主流程不得直接依赖某个具体外部协议
- 订单、物流、售后等能力必须通过统一适配层接入
- 查询结果必须结构化，不能把原始返回直接塞进后续 prompt
- 外部业务事实优先于模型自由生成与 RAG 补充

## 3. 第一阶段建议的接入边界

第一阶段建议把主流程依赖收敛到内部接口，而不是直接依赖：

- MCP 协议细节
- 某个内部 HTTP API SDK
- 某个 CLI 命令格式
- 某个 skill 的原始输出格式

更稳的方式是：

- 主流程只依赖内部 `BusinessDataGateway`
- 不同接入方式通过 adapter 实现

建议的内部能力边界：

- `OrderQueryGateway`
- `LogisticsQueryGateway`
- `AfterSalesPolicyGateway`

如果第一阶段想进一步收口，也可以聚合为：

- `BusinessDataGateway`

但无论采用拆分接口还是统一网关，主流程都不应直接感知下层到底是：

- MCP
- 内部 API
- skill
- CLI

## 4. 第一阶段必须支持的业务查询能力

第一阶段建议至少支持以下查询：

1. 按订单号查询订单状态
2. 按订单号或物流号查询物流进度
3. 查询标准售后规则或标准处理状态说明

第一阶段暂不要求：

- 复杂逆向流程编排
- 自动提交退款或补偿动作
- 跨多个系统串联写操作
- 高风险售后操作自动执行

## 5. 输入合同

业务数据查询层接收的输入，应该来自前面已经收敛好的结构化结果，而不是自由文本。

建议输入字段：

- `customer_id`
- `email`
- `thread_id`
- `scenario`
- `intent`
- `order_id`
- `tracking_number`
- `product_id`
- `query_reason`

要求：

- 如果关键实体缺失，不应盲查
- 应优先返回“缺少必要查询条件”的结构化结果
- 不应把模糊猜测当成实际查询条件发给业务系统

## 6. 输出合同

业务数据接入层必须返回结构化结果。

建议输出字段：

- `query_status`
- `source_system`
- `resolved_entities`
- `business_facts`
- `fact_timestamp`
- `missing_requirements`
- `conflict_flags`
- `error_type`
- `error_message`

字段说明：

- `query_status`：范围建议为 `success / not_found / insufficient_input / temporary_failure / permanent_failure`
- `source_system`：结果来自哪个系统或 adapter
- `resolved_entities`：本次查询最终确认使用的实体
- `business_facts`：结构化业务事实，例如订单状态、物流节点、售后状态
- `fact_timestamp`：事实获取时间
- `missing_requirements`：缺失的必要查询条件
- `conflict_flags`：结果冲突或异常标记
- `error_type`：失败分类

## 7. 查询失败与兜底规则

第一阶段建议明确区分以下情况：

### 7.1 查询条件不足

例如：

- 客户说“我的订单怎么还没到”但没有订单号
- 无法根据邮箱唯一定位订单

处理建议：

- 优先进入“需要追问”
- 一次只追问最关键的标识信息

### 7.2 查询无结果

例如：

- 订单号不存在
- 物流号不存在

处理建议：

- 不要伪造事实
- 回复应说明当前未查到对应记录
- 必要时要求客户补充或核对信息

### 7.3 临时失败

例如：

- 下游系统超时
- MCP 或 API 暂时不可用

处理建议：

- 不应伪装成业务无结果
- 应保留“暂时失败”状态
- 回复链路应进入保守回复、待重试或人工处理

### 7.4 结果冲突

例如：

- 订单系统状态与物流系统状态明显冲突
- 当前系统结果与客户刚提供的新事实冲突

处理建议：

- 标记 `conflict_flags`
- 不强行生成确定结论
- 优先进入人工处理或保守解释

## 8. 第一阶段推荐的 adapter 形态

第一阶段不锁定具体协议，但推荐保留下列 adapter 槽位：

- `OrderQueryAdapter`
- `LogisticsQueryAdapter`
- `AfterSalesPolicyAdapter`

可能的实现来源：

- MCP adapter
- 内部 REST API adapter
- skill adapter
- CLI adapter

这样做的价值：

- 便于先用最容易拿到的能力打通闭环
- 后续替换真实系统时，不需要重写主流程
- 面试展示时也更能体现分层设计能力

## 9. 与上下文和 RAG 的边界

业务数据接入层负责：

- 获取可验证的业务事实
- 产出结构化业务结果

它不负责：

- 最终回复生成
- 产品知识解释
- 政策知识检索增强

边界原则：

- 订单、物流、售后状态属于业务事实层
- 产品说明、常见规则解释、FAQ 说明更适合进入 RAG 或规则知识层

## 10. 第一阶段建议结论

第一阶段应把“外部业务数据接入”视为统一适配层问题，而不是某个协议选型问题。

最小可执行要求是：

- 主流程只依赖内部 gateway 抽象
- 至少支持订单、物流、标准售后规则三类能力
- 结构化返回成功、无结果、条件不足、临时失败、冲突几类状态
- 业务事实与 RAG 知识严格分层
