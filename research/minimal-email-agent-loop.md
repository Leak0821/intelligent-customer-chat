# 邮件 Agent 最小闭环链路研究

## 1. 目的

这份文档用于回答一个更具体的问题：

在已经确定项目范围的前提下，第一条真正值得优先落地的邮件 agent 业务主链路应该长什么样。

这里不解决技术选型，不锁定框架和基础设施，只解决：

- 为什么优先选择“通用主链路 C”
- 第一条最小闭环到底包含哪些节点
- 哪些真实用户体验问题容易被忽略

## 2. 为什么优先选择 C 路径

这里的 `C` 指：

- 先做一个通用主链路
- 先分售前 / 售后两个一级场景
- 每个一级场景下只先支持少量高频子意图
- 后续再逐步叠加更多子意图

相比“只做售前”或“只做售后”的单点闭环，`C` 更适合这个项目当前目标。

### 2.1 更像真实生产系统

真实邮件客服不是单一路径，而是先做主分流，再走不同分支。

如果一开始只做一个孤立用例，后面再扩到售前和售后，通常会重写主流程。

### 2.2 更适合面试展示

面试场景下，单点 demo 容易显得“能跑，但不具备系统性”。

`C` 路径更能体现：

- 业务分层思维
- 主路由能力
- 上下文能力
- 外部查询能力
- RAG 只是补充而不是主真相源
- 人工兜底能力

### 2.3 方便后续按意图逐步扩展

当前阶段不需要一次把所有意图做完，但要确保：

- 主链路稳定
- 新意图可以插入已有流程
- 不会因为新增意图而推翻整体设计

## 3. 推荐的最小闭环主链路

推荐先围绕下面这条业务链路收敛：

1. 收到客户邮件
2. 邮件清洗
3. 提炼或重写用户意图表达
4. 匹配内部一级场景与子意图枚举
5. 加载历史邮件上下文
6. 尝试补充用户画像信息
7. 根据意图调用外部信息源
8. 用 RAG 补充产品、规则或政策知识
9. 生成回复邮件草稿
10. 进入人工审核、自动校验或待发送节点

这条链路的重点不在于每一步做得多复杂，而在于它是完整的、可扩展的、接近真实客服系统的。

## 4. 为什么要加“意图重写 / 归一化”

这一步值得单独写出来，因为它非常像真实生产里的必要层，而不只是一个细节。

原始客户邮件经常有这些特点：

- 语句口语化
- 一封邮件里夹杂多个问题
- 关键实体缺失
- 历史对话粘连严重
- 情绪表达强，但业务意图不清晰

如果只做“清洗后直接分类”，会遇到两个问题：

1. 不利于稳定匹配内部意图枚举
2. 不利于检索、RAG 或工具调用使用统一表达

因此建议主链路里显式保留一层：

- 把用户自然语言问题整理成更标准的内部查询表达
- 再映射到内部意图体系

这层的价值是：

- 提高意图识别稳定性
- 提高检索召回质量
- 降低原始邮件噪声对后续流程的污染

## 5. 第一阶段建议覆盖的子意图风格

不是要现在就冻结完整意图树，而是建议先优先覆盖“高频、结构相对清楚、便于验证”的子意图。

### 5.1 售前建议优先的子意图

- 产品推荐
- 产品差异对比
- 基础功能咨询
- 发货或库存相关咨询

### 5.2 售后建议优先的子意图

- 订单状态查询
- 物流进度查询
- 常见售后规则解释
- 标准化问题回复

### 5.3 第一阶段不建议优先做的子意图

- 高争议投诉处理
- 复杂退款协商
- 高风险补偿承诺
- 跨多个系统才能确认的特殊异常工单

原因不是这些不重要，而是它们通常更依赖人工判断，作为第一阶段闭环会显著提高系统风险。

## 6. 真实用户体验上最容易忽略的点

下面这些点很容易在技术设计时被低估，但在真实客服体验里很关键。

### 6.1 客户不想重复讲一遍自己的问题

如果系统已经有历史邮件、订单、购物车、浏览或既有工单信息，就不应该让客户把这些内容重复一遍。

否则体验会很差，客户会觉得：

- 你没有读懂他的上下文
- 你的自动化只是表面自动化

对这个项目来说，这意味着：

- 上下文加载不是锦上添花，而是基础能力
- 用户画像如果可用，应优先用来减少重复提问

### 6.2 一封邮件可能同时包含多个问题

真实客户经常会把多个问题写在一封邮件里，例如：

- 物流什么时候到
- 如果延迟了是否可以退款
- 顺便再问某个产品是否更适合

因此不要假设“一封邮件 = 一个纯净意图”。

第一阶段至少要考虑：

- 主意图如何确定
- 次意图是否记录
- 当前回合是否只处理主意图，还是同时覆盖多个问题

### 6.3 订单与物流类问题，客户更在意“明确事实”而不是“礼貌废话”

客户问订单或物流时，最重要的是：

- 当前状态是什么
- 下一步是什么
- 预计什么时候有新变化

如果系统只生成一封语气很好但事实不清楚的邮件，体验依然很差。

因此这类场景里：

- 真实业务数据必须优先
- 回复结构要以“事实 + 下一步”优先

### 6.4 售前推荐不要只给一个答案

如果售前推荐只给一个“最优产品”，很容易在信息不充分时显得武断。

更接近真实客服的做法通常是：

- 给 2 到 3 个备选项
- 说明差异和取舍
- 明确当前推荐是基于哪些已知条件
- 如果信息不足，再补一个最小追问

这比“拍脑袋式精准推荐”更可信。

### 6.5 信息不足时，追问比瞎答更像专业客服

很多系统容易犯的错是：

- 看到问题就急着回答
- 不愿承认条件不足

但真实客服里，适当追问本身就是专业能力的一部分。

因此第一阶段就应该明确：

- 什么时候直接答
- 什么时候先追问补信息
- 什么时候直接转人工

### 6.6 审核节点不是多余步骤，而是建立信任的缓冲层

在高风险场景里，人工审核或至少规则校验，不是“效率低”，而是“减少错误承诺”。

对于面试展示也一样：

- 有审核或兜底策略，反而更像能落地的系统
- 没有审核、所有邮件都敢自动发，通常会显得不够稳

### 6.7 需要主动补上自然 follow-up

真实客服邮件里，客户不一定会只问“当前问题”。

好的体验往往来自：

- 你不仅回答当前问题
- 还顺手补了客户下一步大概率会继续问的点

例如：

- 物流未发货时，除了说“还没发”，最好告诉客户下一次可预期更新时间
- 售前推荐后，除了给推荐结果，还可以补一句如果预算或规格不同可继续细化

这不是废话，而是减少来回沟通成本。

### 6.8 客户更能接受“我知道什么 / 我不知道什么 / 下一步怎么办”

比起强行装作全知，更稳的体验通常是：

- 先总结已知事实
- 明确未知部分
- 给出下一步动作

这种回复结构会更可信，也更适合和工具查询、RAG、人工审核结合。

## 7. 第一阶段建议保留的兜底策略

即使暂时不写成最终技术规则，也建议从业务上先保留这些兜底边界：

- 低置信度不强答
- 数据冲突不强答
- 缺订单号或关键实体时可追问
- 投诉升级、补偿承诺、复杂退款先保留人工介入
- 可用业务数据与模型输出不一致时，以业务数据为准

## 8. 对后续 OpenSpec 的启发

如果后续要把第一条闭环正式写成 OpenSpec，建议至少拆成这些主题：

1. 邮件接入与清洗
2. 意图重写与意图枚举匹配
3. 上下文与用户画像加载
4. 售前分支处理
5. 售后分支处理
6. RAG 补充与回复生成
7. 审核、兜底与发送前校验

## 9. 当前建议结论

当前最合理的推进顺序是：

1. 先采用 `C` 作为第一阶段主路径
2. 在主路径里显式加入“意图重写 / 归一化”
3. 一级场景先固定为售前 / 售后
4. 每个一级场景下先落少量高频子意图
5. 从第一版开始就保留审核、兜底和人工接管扩展位

这条路径既不会把项目做成过于简单的 demo，也不会因为一开始范围过大而无法落地。
