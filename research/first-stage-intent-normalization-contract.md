# 第一阶段意图重写 / 归一化合同

## 1. 目的

这份文档用于把“意图重写 / 归一化”从概念层，收敛成第一阶段可以直接实现和验收的合同。

它回答的不是“提示词怎么写”，而是：

- 这一层的输入是什么
- 这一层允许产出什么
- 什么内容可以整理
- 什么内容不能改写
- 什么时候应进入追问
- 什么时候应直接转人工

## 2. 这一层在主链路中的位置

第一阶段主链路中，这一层位于：

`邮件清洗 -> 意图重写 / 归一化 -> 一级场景与子意图映射`

它的职责是把高噪声、非结构化、可能包含多问题的客户邮件，整理成后续路由、检索、工具查询都更容易消费的内部表达。

它不是最终回复生成层，也不是订单或物流真值判断层。

## 3. 输入合同

第一阶段建议这层至少接收以下输入：

- 当前邮件最新一轮客户正文
- 邮件线程中与当前问题直接相关的历史摘要
- 已提取的基础元数据
- 当前已知的客户上下文摘要

建议的输入字段：

- `message_id`
- `thread_id`
- `customer_language`
- `latest_customer_message`
- `thread_summary`
- `known_customer_facts`
- `known_business_entities`

字段说明：

- `latest_customer_message`：当前回合客户最新、待处理的正文
- `thread_summary`：历史往来邮件的压缩摘要，不要求完整原文
- `known_customer_facts`：如历史订单、已有售后记录、已知偏好等
- `known_business_entities`：如订单号、物流单号、产品名、SKU、时间、数量等

## 4. 输出合同

第一阶段建议这层使用结构化输出，而不是自由文本拼接。

建议的输出字段：

- `normalized_request`
- `customer_goal`
- `primary_issue`
- `secondary_issues`
- `scenario_hint`
- `intent_candidates`
- `required_entities`
- `missing_entities`
- `answerability`
- `risk_flags`
- `facts_preserved`
- `assumptions`

字段说明：

- `normalized_request`：归一化后的内部请求表达，用于后续意图识别与检索
- `customer_goal`：客户最想解决的结果，例如“查订单状态”“推荐合适产品”
- `primary_issue`：当前回合主问题
- `secondary_issues`：附带问题列表，第一阶段可只记录不立即处理
- `scenario_hint`：一级场景候选，范围仅限 `售前 / 售后 / 未确定`
- `intent_candidates`：子意图候选列表，允许 1 到 3 个
- `required_entities`：当前主问题需要的关键实体，例如订单号、产品规格、预算、使用场景
- `missing_entities`：当前缺失的关键实体
- `answerability`：范围仅限 `可直接继续 / 需要追问 / 需要人工介入`
- `risk_flags`：高风险或不稳定信号，例如投诉升级、补偿承诺、事实冲突
- `facts_preserved`：明确从原始邮件或上下文中保留下来的事实
- `assumptions`：模型做出的弱推断，必须与事实分开列出

## 5. 允许做的整理动作

这一层允许做以下处理：

- 去掉寒暄、签名、重复表述、明显噪声
- 把口语化表达整理为更清晰的内部问题描述
- 拆分一封邮件中的主问题与附带问题
- 合并同义表达，减少重复
- 从上下文中补全“已明确存在”的事实引用
- 标记缺失信息，而不是硬猜

## 6. 禁止改写的内容

这一层不得做以下事情：

- 不得凭空补造订单号、物流单号、SKU、数量、金额、时间
- 不得把未确认的推测写成事实
- 不得改写客户明确表达的诉求强度
- 不得丢掉可能影响售后判断的关键约束
- 不得擅自承诺退款、补偿、换货、时效
- 不得把历史上下文中的旧事实误写成当前新事实

特别要求：

- 订单号、物流号、SKU、日期、金额、数量、地址、承诺性措辞必须原样保留或显式标记为未知

## 7. 主问题与附带问题规则

第一阶段必须允许“一封邮件包含多个问题”。

当前建议规则：

- `primary_issue` 应代表当前最需要被处理的问题
- `secondary_issues` 记录其余问题，不要求第一阶段同时全部处理
- 如果多个问题都高度关键，且无法安全只处理一个，应把 `answerability` 标记为 `需要人工介入`

主问题优先判断原则：

1. 是否直接影响当前回复动作
2. 是否依赖外部业务数据查询
3. 是否具有明显时效性
4. 是否涉及客户当前最强烈的诉求

## 8. 什么时候进入“需要追问”

以下情况优先标记为 `需要追问`：

- 已能识别大致意图，但缺少完成查询或推荐所需的关键实体
- 客户问题范围过大，需要最小补充问题才能继续
- 售前推荐缺少核心约束，例如预算、用途、规格偏好
- 售后查询缺少订单号、物流号或无法定位订单的关键信息

第一阶段建议采用“最小追问”原则：

- 一次只追问最关键的缺失信息
- 不重复追问系统已经知道的信息
- 不把多个补充问题堆成审问式清单

## 9. 什么时候进入“需要人工介入”

以下情况优先标记为 `需要人工介入`：

- 涉及补偿、退款协商、升级投诉等高风险诉求
- 邮件中存在明显情绪升级或法律风险信号
- 业务数据与邮件内容存在冲突，且无法自动澄清
- 同时存在多个高优先级问题，自动链路难以安全裁剪
- 当前问题超出第一阶段首批子意图覆盖范围

## 10. 第一阶段推荐的输出示例

```json
{
  "normalized_request": "客户想查询订单当前物流进度，并确认如果继续延迟下一步应如何处理。",
  "customer_goal": "获得物流事实状态与下一步处理建议",
  "primary_issue": "物流进度查询",
  "secondary_issues": [
    "延迟后的后续处理方式咨询"
  ],
  "scenario_hint": "售后",
  "intent_candidates": [
    "物流进度查询",
    "常见售后政策解释"
  ],
```

## 11. 与后续节点的边界

这一层完成后，后续节点才负责：

- 一级场景与子意图最终映射
- 调用订单、物流或其他外部系统
- 调用 RAG 做知识补充
- 生成回复草稿
- 审核、校验、发送前处理

因此这层不负责：

- 最终业务决策
- 真实业务事实确认
- 最终回复措辞

## 12. 第一阶段实现建议

当前更推荐：

- 采用“两步走”
- 第一步输出归一化结构
- 第二步再做意图枚举映射

这样做的原因：

- 更容易单独评估归一化质量
- 更容易回放问题邮件并定位错在“重写”还是“分类”
- 更适合后续用 `Spring AI` 或其他方式做结构化输出校验

## 13. 当前建议结论

第一阶段应把“意图重写 / 归一化”视为正式合同层，而不是实现细节。

最小可执行要求是：

- 有明确输入
- 有结构化输出
- 有禁止改写项
- 能区分主问题和附带问题
- 能产出 `可直接继续 / 需要追问 / 需要人工介入` 三类判断
